④ Dep 类与发布订阅（手动订阅）

上一篇我们把 render 写死在 setter 里。问题是：同一个属性可能被多个 UI 用到，
每个 UI 都想在该属性变化时执行自己的副作用——单一回调显然不够。
这一篇引入 Dep（依赖容器）类：每个属性配一个 Dep，里面存订阅者列表。
setter 不再直接调 render，而是调 dep.notify()，逐个通知订阅者。

③ 日志

关键设计：发布订阅
  • Dep：每个属性一个，维护 subs（订阅者数组）
  • addSub(cb)：注册一个回调
  • notify()：依次调用所有订阅者
setter 里 dep.notify() 之后，所有订阅这个属性的回调都会被执行——一对多就实现了。
痛点：手写订阅太繁琐
我们必须明确写出 nameDep.addSub(renderHeader)、nameDep.addSub(renderProfile)……
在真实模板里，{{ name }} 出现在哪里我们都得主动登记一次。如果模板里漏了一处，UI 就不会更新；如果多登记一次，又会无故重复执行。
能不能让"读取过 name"这件事自己被记录下来？——这就是下一篇的精髓。
